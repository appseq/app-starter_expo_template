KEY DESIGN PATTERNS
================================================================================

1. DYNAMIC OBJECT TYPE SUBSTITUTION
   Many strings use template variables for app type flexibility:
   - {objectType} = "rock" (single)
   - {objectTypePlural} = "rocks" (plural)
   
   This allows the app to be adapted for different identification purposes
   (birds, plants, art, etc.)
   
   Examples:
   - "Identify {objectType}..."
   - "Analyzing {objectType}..."
   - "Delete {objectType}?"

2. CENTRALIZED CONFIGURATION
   Single source of truth in constants/appConfig.ts:
   - app.name: "Rock Identifier"
   - ui.tabs: navigation labels
   - ui.camera: camera screen strings
   - ui.results: results screen strings
   - ui.settings: settings strings
   - errors: error messages

3. HIERARCHICAL ORGANIZATION
   UI items follow a consistent pattern:
   - Title (short, memorable)
   - Subtitle (descriptive)
   - Icon (visual indicator)

4. THREE-TIER UI INFORMATION
   Many screens use this pattern:
   1. Title/Header
   2. Description/Subtitle
   3. Action buttons or sections

TRANSLATION MANAGEMENT RECOMMENDATIONS
================================================================================

STEP 1: SELECT TRANSLATION PLATFORM
  Recommended options:
  - Crowdin (professional, integrates with GitHub)
  - Lokalise (user-friendly, good for mobile apps)
  - Phrase (enterprise features)
  - Google Sheets (free/simple for small teams)

STEP 2: PREPARE STRINGS FOR TRANSLATION
  - Use TRANSLATION_STRINGS.csv as import source
  - Provide context notes to translators
  - Specify terminology for geological terms
  - Clarify which strings use dynamic substitution

STEP 3: IMPLEMENTATION IN CODE
  Recommended libraries:
  - react-native-localize
  - i18n-js
  - Expo Localization API
  
  Implementation steps:
  a) Create translation files structure:
     locales/
       ├── en/
       │   ├── common.json
       │   ├── onboarding.json
       │   ├── ui.json
       │   ├── settings.json
       │   └── errors.json
       ├── es/
       ├── fr/
       ├── de/
       └── [other languages]/
  
  b) Create i18n utility:
     - Load locale based on device language
     - Provide fallback (English)
     - Handle language switching

  c) Refactor appConfig.ts:
     - Reference translation keys instead of strings
     - Load translated values at runtime

  d) Handle dynamic substitution:
     - Create template function for {objectType} patterns
     - Test with different object types

STEP 4: TESTING
  - Test RTL languages (Arabic, Hebrew)
  - Verify long string handling (German, Finnish)
  - Check character encoding (Chinese, Japanese, Korean)
  - Test language switching at runtime
  - Verify all strings load correctly

STEP 5: DEPLOYMENT
  - Create language-specific builds or runtime switching
  - Update App Store metadata for new languages
  - Test on real devices with different system languages
  - Consider font support for non-Latin scripts

SPECIAL CONSIDERATIONS
================================================================================

1. GEOLOGICAL TERMINOLOGY
   - Rock and mineral names typically stay in English
   - Technical terms should match regional standards
   - Consider whether to translate category names
   - Verify accuracy with geological experts

2. PLATFORM-SPECIFIC TEXT
   - iOS permission messages should match iOS conventions
   - Alert button text (Cancel, Delete) follows iOS patterns
   - Navigation labels follow iOS tab bar conventions

3. DYNAMIC SUBSTITUTION
   Several strings use dynamic content:
   - App name: {APP_NAME}
   - Object type: {objectType} (rock/bird/plant/etc)
   - Email address: {email}
   - Feature name: {feature}
   - Percentage: {percentage}%
   
   Translators must preserve these placeholders.

4. CONTENT STRUCTURE
   - Onboarding text uses parallel structure (4 slides, same format)
   - Rock education content is scientific (accuracy important)
   - Subscription benefits follow comparison pattern (Free vs Premium)
   - Error messages are concise but helpful

5. BRAND CONSISTENCY
   - App name "Rock Identifier" appears on first onboarding slide
   - Company: "Centriqo" (support email domain)
   - Maintain brand voice across translations
   - Use consistent terminology throughout
